02 / 09

The Challenge

Account and billing information lived in three different places, none of them built for the job.

  • Password hints and PINs scattered across a notes app — no structure, no search.
  • No single view of what's due this week — subscriptions got missed or paid late.
  • No visibility into weak or reused passwords, or how old they were.
  • Client, relatives', and personal logins all mixed together with no way to filter.
  • Paid password managers exist, but mean handing sensitive data to another company's server, for a monthly fee.
03 / 09

The Goal

Build something narrow enough to actually use daily, on tools already in hand.

  • Own the data — store it in a spreadsheet, not a third-party vault.
  • Zero recurring cost — no subscription to track subscriptions with.
  • Two jobs, two views — money and login security shouldn't share one table.
  • Usable on a phone, not just a desktop dashboard.
  • A light gate on the door — not enterprise auth, just enough to stop casual access.
04 / 09

My Approach

Built iteratively, in the order the real gaps showed up.

01Sheets as the database

No hosting, no migrations — a spreadsheet tab as source of truth, editable from a phone.

02Apps Script as the API

Code.gs reads a tab, returns JSON over a token-gated endpoint. No server to maintain.

03Subscription Vault first

KPI cards, a 30-day payment horizon, and a filterable table of every paid account.

04Split, not widened

Login/security data became its own dashboard, sharing one backend via a query param.

05Cross-linked both

One button in each header — two files living as one product.

06Trimmed what didn't earn its place

2FA and Strength dropped; Source Link, PIN, Account Age got promoted.

07Theming and login gate, last

Polish and access control layered onto a working core, not baked in first.

05 / 09

The Solution

A pure static HTML/CSS/JS front end — no framework, no build step — talking to one Apps Script Web App that branches on a query parameter to serve either sheet tab.

Sensitive fields stay masked behind an eye-toggle by default. Every computed figure — password age, account age, days until due — is calculated client-side from raw dates on load, so numbers are correct the moment the page opens.

The Results

  • Two working dashboards, cross-linked, sharing one backend and one login gate.
  • Every password and subscription account previously spread across notes and memory now lives in one filterable, searchable place.
  • Recurring payments visible a full 30 days out, with overdue and due-soon states surfaced automatically instead of discovered late.
  • Zero added monthly cost — the entire system runs on tools already paid for, plus free static hosting.
  • Fully responsive — usable on a phone during an actual bill-paying session, not just at a desk.
07 / 09

Key Takeaways

  • One table trying to do two jobs does neither well. Splitting billing from security made both easier to scan.
  • Let real usage trim the feature list. Fields that looked useful on paper got cut; ones that seemed minor turned out to matter most.
  • One backend, multiple front ends beats duplicating infrastructure — one query param routed two dashboards through one deployment.
  • Compute, don't store, anything derived from a date. Password age and days-to-due are calculated on load, never stale.
  • You don't need a framework to build something real. Static HTML/CSS/JS, shipped as single files, was enough for a daily-use tool.
