How Vault & Ledger Was Built

A plain-language, step-by-step walkthrough of how the two dashboards, the backend, and the deployment fit together — written so you can follow it, explain it, or hand it to someone else to rebuild.

HTML5 CSS3 Vanilla JavaScript Google Apps Script Fetch API LocalStorage Responsive Design
01

What's in the system

Three files do all the work. No framework, no build tools, nothing to install.

  • Code.gs — lives inside the Google Sheet. It's the only piece that ever touches the sheet's data.
  • vault-dashboard.html — the Subscription Vault. Money, plans, due dates.
  • vault-dashboard-emails.html — the Email Vault. Accounts, logins, password hints.

Both dashboards are just single files you can open in any browser. They don't run on a server you have to maintain — they call out to one small script that Google hosts for you, for free.

03

Step 1 — Set up the data layer

Everything starts as rows in a Google Sheet — no database to install. One spreadsheet, two tabs:

  • Vault tab — one row per subscription: account name, category, priority, username, password hint, plan, amount, due date, payment status.
  • Accounts tab — one row per login: platform, category, username, email, recovery email, source link, password hint, PIN, date created, password last updated, notes.

The first row of each tab is just column headers for you to read — the script always starts pulling data from row 2, so headers never get mistaken for a real account.

04

Step 2 — Write the backend (Code.gs)

Apps Script is a small scripting tool built into every Google Sheet — Extensions → Apps Script. It runs Google's own servers, so there's nothing separate to host or pay for. Three things it needs to do: check a password, read the right tab, and hand the data back as JSON (a plain, structured text format browsers understand instantly).

The secret token and sheet names

These live at the very top of the file, in one place, so they're easy to change later:

Code.gs — top of file
const API_TOKEN           = 'YOUR-LONG-RANDOM-SECRET';
const SHEET_NAME          = 'Vault';     // read by vault-dashboard.html
const SHEET_NAME_ACCOUNTS = 'Accounts';  // read by vault-dashboard-emails.html

The token is just a password the script checks before it answers. Anyone who has your Web App URL and this token can read your data — so it's never shared anywhere public, only pasted into your own dashboard files.

The entry point — doGet

Every request to the deployed script runs this one function first. It's the gatekeeper and the router in one:

Code.gs — doGet()
function doGet(e) {
  const p = (e && e.parameter) || {};

  if (p.token !== API_TOKEN) {
    return jsonResponse_({ ok: false, error: 'Unauthorized' });
  }

  const accounts = p.sheet === 'accounts' ? readAccounts_() : readVault_();
  return jsonResponse_({ ok: true, accounts: accounts });
}

Wrong or missing token → it refuses and says so. Right token, no ?sheet=accounts → it reads the Vault tab. Right token with ?sheet=accounts → it reads the Accounts tab instead. That one parameter is the entire difference between the two dashboards' data sources.

Reading a tab into clean data

Each tab gets its own small reader function — same shape, different columns:

Code.gs — readVault_() (shortened)
function readVault_() {
  const sheet = SpreadsheetApp.getActiveSpreadsheet().getSheetByName(SHEET_NAME);
  const lastRow = Math.min(sheet.getLastRow(), MAX_ROWS);
  if (lastRow < 2) return [];

  const values = sheet.getRange(2, 1, lastRow - 1, 20).getValues();
  const out = [];

  values.forEach(function (row) {
    const name = String(row[0] || '').trim();
    if (!name) return; // skip blank rows

    out.push({
      name: name,
      category: String(row[1] || ''),
      username: String(row[3] || ''),
      hint: String(row[4] || ''),
      plan: String(row[11] || ''),
      amount: row[12] === '' ? null : Number(row[12]),
      dueDate: toISO_(row[15])
      // ...remaining columns follow the same pattern
    });
  });

  return out;
}

Each row becomes one plain object. Dates get converted to a simple YYYY-MM-DD text format so the browser can read them with zero guesswork about timezones.

05

Step 3 — Deploy it as a Web App

This turns the script into a real web address that any dashboard can call:

  1. Open the Sheet → Extensions → Apps Script
  2. Paste in Code.gs, save it
  3. Deploy → New deployment → type: Web app
  4. Execute as: Me — it runs with your own Google account's access to the sheet
  5. Who has access: Anyone — needed so the dashboard (running in a plain browser, not logged into Google) can reach it. The token is what keeps it actually private, not this setting
  6. Click Deploy, authorize once, then copy the Web App URL it gives you — it always ends in /exec
  7. Every time Code.gs changes, go back to Manage deployments → edit → New version → Deploy. The URL stays the same; only the logic behind it updates
06

Step 4 — Connect each dashboard to it

Near the top of each HTML file's script, one small block holds the two values that came out of deployment:

vault-dashboard.html — CONFIG block
const WEB_APP_URL = "https://script.google.com/macros/s/XXXXXXXXXXXXXXXXXXXXXXXXXXXXXXXX/exec";
const API_TOKEN   = "YOUR-LONG-RANDOM-SECRET";

That's the whole connection — same two values, pasted into both dashboard files. The only difference between them is one extra piece added to the web address when the Email Vault asks for data:

vault-dashboard-emails.html — the actual fetch
const res = await fetch(`${WEB_APP_URL}?token=${encodeURIComponent(API_TOKEN)}&sheet=accounts`);
const json = await res.json();
rows = json.accounts.map(enrich);

fetch is the browser's built-in way of calling a web address and waiting for an answer — no library needed. enrich() then takes each raw row and calculates the things that shouldn't be stored, like "how many days old is this password" — worked out fresh every time the page loads, from the raw date, so it's never out of date.

What actually shows on screen is plain JavaScript building HTML text from that data (KPI numbers, table rows) and dropping it into the page. CSS then handles all the color, spacing, and the light/dark theme — HTML is the structure, CSS is the look, JavaScript is what fetches the data and keeps it up to date as you search or filter.
07

Step 5 — Theme, login gate, mobile

Three finishing touches, added once the data was flowing correctly:

Light / dark toggle

Theme toggle — shared by both dashboards
function applyTheme(mode){
  document.documentElement.setAttribute("data-theme", mode);
  localStorage.setItem("vault-theme", mode);
}
themeBtn.addEventListener("click", () => {
  applyTheme(document.documentElement.getAttribute("data-theme") === "light" ? "dark" : "light");
});

All the colors are defined once as CSS variables. Flipping data-theme swaps every color at once — localStorage just remembers your choice for next time, on that browser.

A soft login gate

A simple username/password screen sits in front of the Subscription Vault — enough to stop casual access, not meant as bank-grade security:

Login check
if (user === LOGIN_USER && pass === LOGIN_PASS) {
  localStorage.setItem("vault-auth", "1");
  enterApp();
}

Because this check runs in the browser, anyone who opens the page's source code can technically read the password — it's a front door, not a vault door. For a personal tool that's the right amount of security for the effort; a client-facing version would move this check onto the server side instead.

Mobile responsiveness

CSS media queries reshape the layout under certain screen widths — KPI cards stack from a row of eight down to one column, filters go full-width, and the table scrolls sideways instead of squeezing illegibly small. Nothing is hidden on mobile, it just re-flows.

08

Full deployment checklist

  1. Build the sheet — two tabs, headers in row 1, data from row 2
  2. Write Code.gs — token, sheet names, doGet, one reader function per tab
  3. Deploy as Web App — Execute as Me, Access: Anyone, copy the /exec URL
  4. Paste URL + token into the CONFIG block of both HTML files
  5. Open the HTML files directly, or drop them on any static host (Netlify, GitHub Pages, a plain folder) — no server, no build step
  6. Re-deploy a new version in Apps Script any time Code.gs changes — the URL never changes, only what's behind it
